# 2026 年度战略规划与执行方案 (Strategic Execution Plan)

> **版本**: v1.0
> **日期**: 2026-01-09
> **目标**: 基于 Architecture 2.0，稳步推进 TinyML 与双语言引擎(C++/MicroPython)的落地。

---

## 1. 核心战略图 (The Big Picture)

我们将 2026 年的开发划分为三个战役阶段：

| 阶段 | 关键词 | 核心目标 | 关键产出 |
| :--- | :--- | :--- | :--- |
| **Q1 (Jan-Mar)** | **筑基 (Foundation)** | **生成器架构重构** | 抽象工厂生成器 (Abstract Generator Factory), Stable v2.0 发布 |
| **Q2 (Apr-Jun)** | **突破 (Breakthrough)** | **TinyML 落地** | 能够识别手势的 Magic Wand 积木, TFLM 引擎集成 |
| **Q3 (Jul-Sep)** | **扩张 (Expansion)** | **MicroPython 兼容** | STM32/ESP32 双语言支持, 烧录工具链集成 |
| **Q4 (Oct-Dec)** | **云化 (Cloud)** | **Web Lite 原型** | 纯网页版 IDE, WebSerial 烧录 |

## 2. 详细执行路线 (Execution Roadmap)

### 🔴 Phase 1: 架构筑基 (Q1 - The "Must Have")
*没有任何新功能比“可维护性”更重要。在引入 TinyML 和 Python 之前，必须先改造地基。*

*   **1.1 [重构] 抽象积木工厂 (Generic Block Factory)**
    *   **现状**: `arduinoGenerator.forBlock[...] = "code string"` (硬编码 C++)
    *   **目标**: `BlockLogic.compile(target: 'cpp' | 'py')`
    *   **行动**:
        *   定义 `IGenerator` 接口。
        *   重写核心积木（Logic, Loops, Math）以符合新接口。
        *   **验证**: 单元测试确保重构前后的 C++ 代码完全一致。

*   **1.2 [架构] 外部工具链适配器 (External Toolchain Adapter)**
    *   **目标**: 为 STM32 烧录做准备。
    *   **行动**:
        *   在 `PioService` 之外封装 `FlashToolService`。
        *   实现对 `st-flash` (STM32) 和 `esptool` (ESP32) 的 CLI 调用封装。

*   **1.3 [Architecture] 全平台芯片自动发现架构 (Unified Board Auto-Discovery)**
    *   **愿景**: 实现 "PIO 更新，软件即更新" (PIO Updates = Software Updates)。建立全平台的自适应硬件生态。
    *   **方案**: **统一数据挖掘脚本 (Unified Data Mining Script)**。
        *   **数据源**: `pio boards [platform] --json-output` (覆盖 `atmelavr`, `espressif32`, `ststm32`)。
        *   **分级策略 (Hierarchy Strategy)**:
            *   **Arduino (AVR)**: 单层列表 (e.g., Uno, Nano, Mega)。
            *   **ESP32**: 单层/双层列表 (e.g., ESP32-S3, C3, Standard)。
            *   **STM32**: **三级架构** (Platform -> Series [F1/F4] -> Variants [C8/ZE])。
        *   **自动化**: 构建时自动运行 `scripts/scan_all_boards.ts`，生成 UI 所需的层级化 JSON 配置。

*   **1.4 [UX] 基于能力的动态积木箱 (Dynamic Toolbox)**
    *   **问题**: 用户在不支持 WiFi 的 Arduino Uno 上看到了 WiFi 积木，导致编译失败或困惑。
    *   **方案**: **能力驱动显隐 (Capability-Driven Visibility)**。
    *   **原理**: 自动挖掘脚本现已包含 `capabilities` 字段 (e.g., `['wifi', 'can', 'bluetooth']`)。
    *   **实现**: 前端 Toolbox 加载时检查当前板卡的 Capabilities。如果 `!capabilities.includes('wifi')`，自动隐藏 Network 类目。
    *   **价值**: 让编辑器“变聪明”，只显示硬件支持的功能。

### 🟡 Phase 2: AI 攻坚 (Q2 - The "Killer Feature")
*让 EmbedBlocks 成为市面上最易用的 TinyML 入门工具。*

*   **2.1 [硬件] 官方套件定型**
    *   **选型**: ESP32-S3 (高性能+AI指令集) + MPU6050 (六轴传感器)。
    *   **产出**: `docs/hardware_standard.md` (标准引脚定义)。

*   **2.2 [引擎] TFLM 静态库集成**
    *   **难点**: 编译 `tensorflow-lite-micro` 极其耗时。
    *   **方案**: 预编译为 `.a` 静态库，作为 PlatformIO 的 `lib_deps` 引入。

*   **2.3 [积木] 数据采集与训练闭环**
    *   开发 `Serial Data Sampler`（PC端采集工具）。
    *   集成 Edge Impulse API（可选）或实现本地 KNN 算法积木。

### 🟢 Phase 3: 双核驱动 (Q3 - The "Ecosystem")
*正式引入 MicroPython，通过 STM32 打开教育存量市场。*

*   **3.1 [后端] MicroPython Service**
    *   实现 `REPL` 终端交互（基于 xterm.js）。
    *   实现 `FileManager` (基于 WebSerial/Ampy) 用于文件上传。

*   **3.2 [内容] STM32 专属支持包**
    *   发布 `stm32-micropython-pack` 插件。
    *   包含：固件烧录向导 (GUI for st-flash) + 专用积木库。

## 3. 立即行动清单 (Immediate Actions - Next 2 Weeks)

为了启动 Q1 计划，请团队关注以下任务：

1.  **技术验证 (POC)**:
    *   **任务**: 选一个最复杂的积木（如 `servo_write`），尝试用 **抽象工厂模式** 重写其生成逻辑，同时输出 C++ 和 Python 伪代码。
    *   **目的**: 验证“双生成器”架构的复杂度是否可控。

2.  **工具链测试**:
    *   **任务**: 编写一个简单的 Node.js 脚本，测试在 Windows 下调用 `st-flash.exe` 能否成功给 STM32F4 刷入 `.bin` 文件。
    *   **目的**: 扫清 Q3 STM32 支持的所有技术障碍。

3.  **TinyML 物料**:
    *   采购 5 套 ESP32-S3 + MPU6050 开发板，分发给核心开发人员。

---

**结语**: 这是一个“先苦后甜”的计划。Q1 的架构重构可能会比较枯燥，但一旦完成，Q2/Q3 的新功能（AI、Python）将如雨后春笋般涌现。
