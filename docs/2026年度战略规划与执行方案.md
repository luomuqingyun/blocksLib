# 2026 年度战略规划与执行方案 (Strategic Execution Plan)

> **版本**: v1.0
> **日期**: 2026-01-09
> **目标**: 基于 Architecture 2.0，稳步推进 TinyML 与双语言引擎(C++/MicroPython)的落地。

---

## 1. 核心战略图 (The Big Picture)

我们将 2026 年的开发划分为三个战役阶段：

| 阶段 | 关键词 | 核心目标 | 关键产出 |
| :--- | :--- | :--- | :--- |
| **Q1 (Jan-Mar)** | **筑基 (Foundation)** | **生成器架构重构** | 抽象工厂生成器 (Abstract Generator Factory), Stable v2.0 发布 |
| **Q2 (Apr-Jun)** | **突破 (Breakthrough)** | **TinyML 落地** | 能够识别手势的 Magic Wand 积木, TFLM 引擎集成 |
| **Q3 (Jul-Sep)** | **扩张 (Expansion)** | **MicroPython 兼容** | STM32/ESP32 双语言支持, 烧录工具链集成 |
### � Phase 4: Pro 创新 (Q4 & Beyond - The "Innovation")
*引入 Rust/Embassy 技术栈，将 EmbedBlocks 提升为生产级安全代码生成工具。*

*   **4.1 [架构] 多语言生成引擎 (Multi-Language Engine)**
    *   **愿景**: 实现 "一次拖拽，多语言输出" (One Block, Multiple Languages)。
    *   **策略**:
        *   **C/C++ (Core)**: 现有的 Arduino/PlatformIO 基础，保持稳定与兼容。
        *   **Python (Expansion)**: 面向教育市场，集成 MicroPython/CircuitPython。
        *   **Rust (Frontier)**: 面向高阶开发，引入 **Embassy** 框架，提供内存安全与异步特性。

*   **4.2 [特性] Embassy 灵感落地**
    *   **可视化异步 (Visual Async)**: 引入 `Async Task` 积木，在 C++ 层利用协程/ProtoThreads，在 Rust 层直接映射 `async/await`，彻底解决多任务编程痛点。
    *   **实时资源冲突检测 (Visual Borrow Checker)**: 借鉴 Rust 所有权思想，在积木拖拽阶段实时检测引脚冲突（如 SPI SCK 与 LED 由于引脚复用导致的冲突），并在 UI 上置灰不可用选项。

## 3. 立即行动清单 (Immediate Actions - Next 2 Weeks)

为了启动 Q1 计划，请团队关注以下任务：

1.  **Phase 4: 硬件动态积木能力落地 (Capability Logic)**:
    *   **任务**: 选一个标准积木（如 `analog_read`），尝试修改其 Block 定义，使其下拉菜单能够直接读取当前板卡的 `pin_options.analog` 数组。
    *   **目的**: 验证离线芯片数据能否完美驱动 UI 积木的动态配置。

2.  **STM32 硬件库深度封装 (Hardware Library Wrap)**:
    *   **任务**: 基于 `CodeBuilder` 引擎，为 STM32 常见的 ADC/PWM/UART 编写更具语义化的积木，取代泛用的 Arduino 积木。
    *   **目的**: 进一步发挥官方数据（100% 物理引脚覆盖）的优势，提升专业感。

3.  **技术验证 (POC) - 双生成器架构**:
    *   **任务**: 选一个最复杂的积木（如 `servo_write`），尝试用 **抽象工厂模式** 重写其生成逻辑，同时输出 C++ 和 Python 伪代码。
    *   **目的**: 验证“双生成器”架构的复杂度是否可控，为 Q3 引入 MicroPython 铺路。

---

**结语**: 这是一个“先苦后甜”的计划。Q1 的架构重构可能会比较枯燥，但一旦完成，我们将拥有一个能够同时支撑 **C++ 生态 (稳定)**、**Python 生态 (易用)** 和 **Rust 生态 (安全)** 的强大图形化编程平台。2026，我们将重新定义嵌入式低代码开发。
