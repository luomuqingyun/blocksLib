### Pattern 5: 状态感知型智能触发 (Smart Drag-Aware Triggering)

**场景**: 在重型组件（如 Blockly）中，用户的拖拽操作会产生每秒上百次的事件流，如果每次都触发生成代码或扫描变量，会导致明显的 CPU 噪声和误判。

**实践**: 
1. **区分操作状态**: 监听 `DRAG_START/END` 以及 `WidgetDiv.isVisible()` (输入框可见性)。
2. **静默 (Silence)**: 操作进行中（如拖拽、连续打字）挂起昂贵的计算。
3. **即时 (Immediate)**: 操作结束（如积木落地、输入框失焦）瞬间执行计算。

**收益**: 
- **零消耗**: 操作过程中几乎不占 CPU。
- **高反馈**: 结果呈现没有“迟滞感”。
- **一致性**: 避免了因为积木处于悬空/断开状态而产生的中间态逻辑报错。

---

### Pattern 6: 领域逻辑 Hook 化 (Hook Extraction)

**场景**: 当 React Context 变得过于臃肿（超过 500 行），包含大量副作用和底层逻辑时。

**实践**: 
1. 将业务操作（如文件 IO、串口通讯协议）提取到独立的 Custom Hooks 中。
2. Context 仅保留最小状态集和 Provider 分发。

**示例**:
- `FileSystemContext` -> `useProjectOps`, `useAutoBackup`
- `SerialContext` -> `useSerialMonitor`, `useSerialActions`

**收益**: 
- **高内聚**: 逻辑逻辑按领域物理隔离。
- **易测试**: 纯逻辑 Hook 相比 Context 及其 UI 绑定更易于单元测试。
- **可复用**: 相同逻辑可在不同组件或页面中快速复用，无需依赖庞大的 Context。

### Pattern 7: 编辑器状态同步稳定性 (Editor State Stability) [2026-02-04]

**现象**: 在全量更新文本内容（如 `setValue`）时，编辑器发生非预期的滚动跳转，甚至在窗口尺寸未变时也发生抖动。

**机理**: 
1. **渲染链抖动**: React 重绘期间父容器可能产生微秒级的“无效尺寸”，触发了 `automaticLayout` 的重对齐。
2. **全量重置**: `setValue` 默认会尝试重置视图状态，若新旧内容行数差异大，虚拟滚动锚点会失效。
3. **边界模糊**: `scrollBeyondLastLine: true` 会导致文档底部存在动态生成的空白区，干扰定位算法。

**实践**: 
1. **引用稳定化**: 使用 `useRef` 隔离 Setter 和代码引用，防止打字时触发无关的 React Reconciliation。
2. **物理边界约束**: 设置 `scrollBeyondLastLine: false`，让滚动的逻辑边界与真实的物理行号强绑定。
3. **锁保护**: 在内容初始化加载期间使用 `isLoading` 锁，屏蔽所有不稳定的触发源。

### Pattern 8: 工业级加载锁 (Initialization Loading Guard) [2026-02-04]

**场景**: 异步加载重型数据（如 Blockly XML、大型 JSON）时，初始化过程中的中间态（如空工作区）可能被错误识别为“用户修改”并触发持久化（保存/备份），导致数据被清空。

**实践**: 
1. **全局状态锁**: 在 `FileSystemContext` 引入 `isLoading`。
2. **物理拦截**: 在保存函数、自动备份 Hook 顶部强制检测 `isLoading`。如果锁开启，直接拦截写操作。
3. **事件静默**: 在加载数据前执行 `Blockly.Events.disable()`，加载后再启用。

**收益**: 
- **100% 安全**: 彻底杜绝初始化竞态导致的代码丢失。
- **鲁棒性**: 确保只有在“就绪且稳定”的状态下，才允许进行磁盘 IO。

### Pattern 9: 模式感知的事件解耦 (Mode-Aware Event Decoupling) [2026-02-10]

**场景**: 在一个受多维状态驱动的重型 React 组件中，某些状态变更（如工作区版本更新）会导致不必要的重绘或逻辑重置，特别是当这些变更与当前所处的交互模式（如工具箱搜索）无关时。

**实践**: 
1. **定义对照引用**: 使用 `useRef` 记录上一次驱动逻辑的关键参数（如 query, mode, isVisible）。
2. **建立拦截条件**: 在 `useEffect` 顶部通过逻辑与/或判断变动的纯粹性。
3. **精准阻断**: 如果检测到是无关信号触发，直接 `return`。

**收益**: 
- **极高的稳定性**: 保护了 UI 的当前交互状态（如列表滚动位置、选中索引）不被外部状态脉冲破坏。
- **性能红利**: 物理切断了无效的重算链。

### 11. 交互态独占保护原则 (Interactivity Exclusivity Pattern) [2026-02-10]
**场景**: 面对无法复现的“灵异”输入 Bug，单纯的代码静态分析难以覆盖浏览器底层焦点切换冲突。
**实践**: 
1. **方案 1 (延迟避让)**: 差异化 `setTimeout` 策略，打字期采用长延迟 (800ms+)，结构变动采用短延迟 (200ms)。
2. **方案 2 (物理拦截)**: 在执行任何破坏性 DOM 操作（如 `setWarningText`）前，断言 `!block.isEditing()`。
3. **方案 3 (状态解耦 - 进阶)**: 若业务极速复杂，建议将状态外迁至 Redux/Zustand 等中间层，实现 UI 与逻辑的完全物理隔离。

### 12. 语义化冲突归属模式 (Trigger-Aware Attribution Pattern) [2026-02-10]
**场景**: 多对象互斥（引脚重复、名称冲突）时，简单的“首位命中”会导致不合逻辑的警告指向。
**实践**: 
1. **变更采样**: 在事件循环起始处通过 `event.blockId` 提取活跃对象。
2. **责任优先**: 校验算法内部对比：`if (currentBlock !== trigger && otherBlock === trigger) skip;`。
3. **收益**: 让用户清晰地感知到“是我的哪次操作引发了系统的保护机制”，显著降低认知负荷。


### 13. 核心单例资产保护 (Singleton & Undeletable Protection) [2026-02-10]
**场景**: 每个项目必不可少的根节点（如 Setup/Loop 入口）或全局单例配置块。
**实践**: 
1. **多重防护**: 结合 `setDeletable(false)`、`duplicatable = false` 与 `maxInstances: 1`。
2. **逻辑兜底**: 在校验规则中增加单例冲突检测，引导用户处理手动粘贴导致的冗余块。
3. **收益**: 极大地降低了新手用户因误删或克隆核心块而导致的“不可挽回型”损失。

### 14. 资源感知与 UI 状态解耦 (Decoupling Resource Perception from UI State) [2026-02-10]
**原则**: 硬件资源（引脚、定时器、串口）的占用逻辑应与积木的 UI 激活状态（Enabled/Disabled）解耦。
- **坑点**: 若校验规则在检查冲突时跳过“已禁用”的积木，会导致校验系统进入“自我修复”的循环谬误（因为报错会导致禁用，禁用会导致冲突消失，冲突消失会导致重新启用，重新启用又会导致报错）。
- **实践**: 硬件冲突检测必须扫描工作区内所有处于 `isInFlyout === false` 状态的积木，无论其当前是否被禁用。

### 15. 核心注册表的状态保护模式 (Registry State Protection) [2026-03-05]
**场景**: 在具备动态加载/异步背景任务（如 STM32 主库后台加载）的系统中，初始化逻辑可能会覆盖由外部扩展（插件）在运行时动态注入的数据。
- **坑点**: 若 `initialize()` 函数盲目执行 `clear()`，会由于异步时序问题（Race Condition）导致已加载的扩展数据“瞬间消失”。
- **实践**: 在全局注册表刷新时，通过 ID 特征（如插件命名空间 `:`）区分内置资产与第三方资产。仅清除受内置仓库管理的 ID，保留来自外部扩展的活跃资源。
- **收益**: 杜绝了由于复杂后端加载导致的 UI 组件数据断档，确保了插件系统与主程序核心数据源的共存稳定性。

### 11. 诊断系统的演进与清理机制 (Diagnostic Evolution & Cleanup) [2026-02-10]
**原则**：开发期的诊断工具在被更高效的“后台服务模式”取代后，应立即从 UI 树中剥离。
- **实践**：移除了手动切换的 `DiagnosticOverlay` (V1)，仅保留通过全局快捷键触发的文件导出逻辑 (V2)。这降低了主进程与渲染进程间的无效 IPC 往返。
