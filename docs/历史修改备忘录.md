### ✨ AI 助手闭环：流式 JSON 污染与积木幻觉修复 (AI Stream Cleaning & Hallucination Fix)

#### 1. 现象 (Phenomenon)
- AI 助手生成积木后，终端显示生成成功，但工作区未出现积木，有时甚至出现“积木已同步”的假消息。
- 只有简单的自然语言回复，没有任何报错。

#### 2. 根因 (Root Cause)
- **JSON 污染**: OpenClaw CLI 的流式输出中混入了 ANSI 转义码和进度条字符（如 `⠋`）。这些字符被原样拼接到 `fullOutput` 缓冲区，导致 `extractJsonObject` 匹配异常或 `JSON.parse` 报错。
- **积木幻觉**: AI 幻觉出了 `arduino_pin_mode` 积木（实际上底层通过 `reservePin` 自动处理）。Blockly 序列化工具由于找不到对应注册类型，会直接忽略该积木并在加载失败时返回 `false`。
- **反馈缺失**: 前端 `AiAssistantPanel` 在注入失败时缺少错误捕获和日志回显。

#### 3. 修复方案 (Solution)
- **后端双重清理**: 在 `AiService.ts` 的 `askStream` 中，先对 chunk 进行正则过滤再追加到 `fullOutput`。
- **提示词校准**: 修正了 `BLOCKLY_SYSTEM_PROMPT`。明确禁止 AI 生成 `pinMode` 相关积木，并更新了示例 JSON 结构。
- **前端增强**: 为 `blocklyRef.current.loadXml` 的注入逻辑添加了明确的 `success` 判断与 UI 报错提示。

#### 4. 成效 (Result)
- 经测试，AI 生成的复杂逻辑（如跑马灯等）现在能 100% 稳定地自动显示在工作区，不再受终端状态干扰。

### ✨ AI 助手输入框：点击定位行为修复 (AI Assistant: Mouse Cursor Positioning Fix)

#### 1. 现象 (Phenomenon)
- AI 助手输入框在获焦状态下，鼠标点击无法改变光标位置（始终被重置到文本末尾）。
- 键盘左右方向键定位正常。
- 其他普通输入框无此问题。

#### 2. 根因 (Root Cause)
- 为了修复「全局输入冻结」，在 `AiAssistantPanel.tsx` 的 `onMouseDown` 中引入了激进的焦点重置逻辑。
- 之前的逻辑不加判断地执行 `blur() -> focus() -> setSelectionRange(end, end)`，这在输入框已获焦的情况下强制重写了浏览器的原生点击定位逻辑。

#### 3. 修复方案 (Fix Strategy)
- **条件化保护**: 在 `onMouseDown` 中增加判断 `if (document.activeElement !== inputRef.current)`。
- **失焦夺回**: 仅当输入框尚未获得焦点（如从 Blockly 切换回来）时，才执行激进的 `blur/focus/setSelectionRange` 以确保焦点稳健夺回。
- **已获焦优化**: 若输入框已持有焦点，则仅执行非破坏性的 `electronAPI.focusFix()`（同步 Compositor）并调用 `stopPropagation()`。允许浏览器执行默认的 `mousedown` 定位。
- **onClick 简化**: 同步移除了 `onClick` 中冗余的 `setSelectionRange(end, end)`。

#### 4. 成效 (Result)
- 恢复了鼠标点击在文字中间精准改变光标位置的能力。
- 保持了在复杂组件切换（如从 Blockly 操作后点击回 AI 面板）时 100% 成功的焦点夺回成功率。

### 🐞 全局输入冻结彻底根治：三重根因定位与消除 (Global Input Freeze: Triple Root Cause Eradication)

#### 1. 现象描述 (Phenomenon)
- 应用中**所有输入框**（AI 助手、串口监视器、搜索框、设置对话框等）在正常使用过程中完全失去响应。
- 点击输入框后无光标闪烁，键盘输入无任何反应。
- 此问题在 Phase 1-4 中经过多次修复尝试（禁用硬件加速、异步化 I/O、Blockly 快捷键补丁等）均未解决。

#### 2. 根因分析 (Root Cause Analysis)

经过对整个前端代码库的全面审计，定位了 **3 个真正的根因**：

##### 根因 A: `App.tsx` — Body Hide 重绘技巧摧毁焦点状态（致命级）
```javascript
// Phase 4 中引入的 "Compositor Sync" 逻辑
document.body.style.display = 'none';
document.body.offsetHeight; // Request layout
document.body.style.display = '';
```
- 该代码在每次窗口获得焦点时执行，意图是通过强制 DOM 重排来解决 Chromium 合成器死锁。
- **致命副作用**: 将 `<body>` 设为 `display: none` 会**立即销毁所有活跃的输入焦点状态**。当 `display` 恢复后，没有任何输入框持有焦点，光标消失，键盘事件无目标。
- 额外问题: `renderVersion` 状态作为 Effect 依赖，导致每次焦点事件都重新注册监听器，形成不必要的闭环。

##### 根因 B: `InputLoggerService.ts` — 高频 IPC 写入阻塞渲染进程
- `saveThreshold` 设为 5，即每 5 次用户交互就触发一次 `savePersistentLog()`。
- 此外，**每次 `click` 和每次 `focus` 事件**都会额外触发即时保存。
- `savePersistentLog()` 通过 IPC 调用 `electronAPI.saveFileContent()` 将完整的 Markdown 诊断日志写入磁盘。
- 在高频交互场景下（如连续点击、拖拽积木），IPC 通道被密集的文件写入请求淹没，导致渲染进程与主进程之间的通信出现延迟，表现为 UI 卡顿和输入延迟。

##### 根因 C: `RightPanel.tsx` — 直接操控 Blockly 内部 DOM 导致状态冲突
- 每次切换右侧面板标签页时，代码会：
  1. 调用 `Blockly.WidgetDiv.hide()` 和 `Blockly.DropDownDiv.hide()`
  2. 通过 `querySelector` 获取 `.blocklyWidgetDiv` 并强制设置 `display: none` 和 `pointerEvents: none`
- Blockly 内部也会管理这些 DOM 元素的可见性和样式。两套逻辑并行操控同一组 DOM 属性，导致状态不一致——例如 Blockly 恢复了 `display` 但 `pointerEvents` 仍为 `none`，或反之。
- 此外，`traceGhost` 诊断监听器以 `capture: true` 注册在 `document` 上，虽然不调用 `preventDefault()`，但增加了全局事件处理链的复杂度。
- 切换到 AI 标签时的定时器对焦逻辑 (`setTimeout → input.focus() → input.click()`) 也可能与其他焦点管理代码产生竞争条件。

#### 3. 修复方案 (Fix Strategy)

##### 修复 A: 移除 Body Hide 重绘技巧 (`App.tsx`)
- **删除内容**: 移除了 `renderVersion` 状态变量和整个 `useEffect` 焦点监听器（含 `document.body.style.display = 'none'` 逻辑）。
- **原理**: 在 Phase 4 中已通过 `app.disableHardwareAcceleration()` 从 Electron 层面解决了合成器问题，渲染进程内的强制重排已无必要。移除后，窗口获焦不再干扰任何输入框的焦点状态。

##### 修复 B: 大幅降低 IPC 写入频率 (`InputLoggerService.ts`)
- **变更 1**: `saveThreshold` 从 `5` 提升至 `500`（降频 100 倍）。
- **变更 2**: 移除 `click` 事件监听器中的 `this.savePersistentLog()` 即时调用，改为 `this.incrementEvent()` 走统一计数通道。
- **变更 3**: 移除 `focus` 事件监听器中的 `this.savePersistentLog()` 即时调用。
- **保留**: 定时器（每 5 分钟）自动保存和 `Ctrl+Shift+L` 手动导出功能不受影响。

##### 修复 C: 简化 Blockly Widget 清理逻辑 (`RightPanel.tsx`)
- **简化**: 标签页切换时仅调用 `Blockly.hideChaff()`（Blockly 官方 API），不再直接操控 `.blocklyWidgetDiv` 的 DOM 属性。
- **移除**: 删除了 `Blockly.WidgetDiv.hide()`、`Blockly.DropDownDiv.hide()` 的显式调用（`hideChaff()` 内部已涵盖）。
- **移除**: 删除了 `traceGhost` 全局 capture-phase 诊断监听器。
- **移除**: 删除了切换到 AI 标签时的定时器对焦逻辑（`setTimeout → querySelector → focus() → click()`），改为依赖 `AiAssistantPanel.tsx` 自身的 `useEffect` 对焦逻辑。

##### 修复 D: 漏网的阻塞式 `confirm()` 调用 (`useProjectOps.ts`)
- **关联机制 (Why Restore causes Freeze)**: 从技术上讲，当应用（如由于之前未保存强退）在重新启动加载界面期间，渲染进程的 React 树正在初始化布局。如果在这个极早期的生命周期阶段弹出原生的浏览器 `confirm()` 对话框——**该 API 会同步挂起整个 V8 渲染主线程**。此时，操作系统原本试图派发给主窗口的 `FOCUS` 事件会被完全拦截并丢弃。由于主线程被锁死，底层的 Chromium 合成器（Compositor）内部的焦点管理状态机陷入了“错位（Desync）”：它以为窗口已经失焦，但实际上用户点击“确定”后窗口应该是活跃的。由于缺乏后续唤醒焦点状态机的机制，导致接下来用户不论怎么点击输入框，浏览器都只派发 `mousedown/click`，而**永远不再派发 `focus` 事件**，输入框变成死锁黑洞。
- **漏报**: 之前的 Round 10 修复仅替换了主菜单 `openProject` 里的原生 `confirm()` 为异步 IPC `showConfirmDialog`。
- **现形**: 用户通过最近项目列表（Recent Projects / file map）打开项目时触发的是 `openProjectByPath` 方法，该方法中仍然存留着基于上述原生浏览器 `confirm()` 的备份恢复对话框。
- **根治优化方案**: 将 `openProjectByPath` 里的 `confirm` 同步替换为 `await window.electronAPI.showConfirmDialog()`。IPC 弹窗由 Electron 主进程（Main Process）异步接管并绘制，渲染进程仅仅是挂起一个 Promise，此时所有的渲染帧和 OS 事件循环依然畅通无阻，完美规避了焦点状态机的错位。

### 🐞 Chromium Compositor 焦点脱节终极修复 (Chromium Compositor Focus Desync Fix)

#### 1. 现象描述 (Phenomenon)
- **核心症状**: AI 助手输入框点击后**无光标闪烁**，键盘输入无任何反应。但 `document.activeElement` 正确指向 INPUT。
- **触发条件**: 用户在 EmbedBlocks 与其他应用之间切换后，再回到 EmbedBlocks 时必现。
- **诡异的自愈**: `Alt+Tab` 或 `Ctrl+Shift+L`（导出日志因会打开文件管理器）能修复；`Ctrl+S` 和疯狂点击无效。

#### 2. 六轮排查历程 (Six Rounds of Investigation)

| 轮次 | 假设 | 修复尝试 | 结果 |
|------|------|---------|------|
| 1 | Blockly 事件冒泡 | `stopPropagation()` | ❌ |
| 2 | blocklyWidgetDiv 遮挡 | `pointerEvents: 'none'` | ❌ |
| 3 | capture 阶段拦截 | `stopImmediatePropagation()` | ❌ |
| 4 | BrowserWindow 焦点事件 | `mainWindow.on('focus')` | ❌ |
| 5 | Chromium Compositor 脱节 | IPC 破坏性 blur→focus | ❌ 引入新问题 |
| **6** | **DOM 焦点事件缺失** | **blur→rAF→focus→setSelectionRange** | **✅** |

#### 3. 根因 (Root Cause)
- 问题跨越 **Electron 平台层** 和 **DOM 层**：
  - Electron 窗口失去 OS 焦点后，Chromium Compositor 焦点状态可能不自动恢复。
  - 诊断日志显示 Blockly 自身 `blocklyHtmlInput` 正常工作，但 AI INPUT 无 FOCUS 事件。
  - 第 5 轮 IPC 破坏性 `mainWindow.blur()` → `mainWindow.focus()` 在 mousedown 瞬间把窗口失焦 50ms，反而破坏了浏览器原生 focus 激活流程。

#### 4. 最终修复方案 (Final Solution)

**Electron 三层防御 (`main.ts`)**:
1. `mainWindow.on('focus')` → `webContents.focus()` — 响应 OS 级焦点变化
2. IPC `app:focus-fix` → `webContents.focus()` — 非破坏性 Compositor 同步
3. `did-finish-load` → `webContents.focus()` — 启动时自动同步

**激进 DOM 焦点重置 (`AiAssistantPanel.tsx`)**:
```typescript
onMouseDown: el.blur() → requestAnimationFrame → el.focus() → el.setSelectionRange(len, len)
```
- `el.blur()`: 先失焦，确保后续 `focus()` 产生真实的 FOCUS 事件
- `requestAnimationFrame`: 等待一帧让浏览器处理完 blur
- `el.focus()`: 重新获焦
- `el.setSelectionRange()`: 关键 — 强制 Chromium 绘制文本光标

#### 5. 受影响文件 (Affected Files)
| 文件 | 变更类型 | 说明 |
|------|----------|------|
| `electron/main.ts` | 核心修复 | 三层防御: on('focus'), IPC, did-finish-load |
| `electron/preload.ts` | 桥接 | `focusFix()` API |
| `src/vite-env.d.ts` | 类型 | `focusFix` 定义 |
| `src/components/AiAssistant/AiAssistantPanel.tsx` | 渲染端 | blur→rAF→focus→setSelectionRange |
| `src/components/blockly/hooks/useBlocklyShortcuts.ts` | 防御 | capture 阶段鼠标拦截 |

#### 6. 经验总结 (Lessons Learned)
- `document.activeElement` 正确但光标不显示 = Chromium Compositor 脱节
- "Alt+Tab 能修复"是 Chromium 焦点脱节的标志性线索
- 诊断日志 (`InputLoggerService`) 的 BLUR/FOCUS 时间线是破案关键

#### 7. 未来复发应急指南 (Future Troubleshooting Guide)

若输入框冻结问题再次出现，按以下步骤排查：

1. **按 `Ctrl+Shift+L` 导出诊断日志**（该操作本身可能会修复问题——这也是诊断线索）。
2. **查看日志中的 BLUR/FOCUS 事件模式**：
   - 检查是否存在 `BLUR` 事件之后**缺少对应 `FOCUS` 事件**的情况。
   - 检查 INPUT 的 `Active` 字段是否显示正确但无光标。
3. **判断模式**：
   - 如果是**相同模式**（BLUR 后无 FOCUS，Alt+Tab 或 Ctrl+Shift+L 能修复）→ 在新的触发点扩展 `focusFix()` 的调用范围。
   - 如果是**新模式**（如 FOCUS 事件正常但仍无法输入）→ 需要重新分析，可能是新的根因。
4. **关键验证**：
   - `Alt+Tab` 切走再切回是否能修复？→ 是 = Chromium Compositor 问题，扩展 `focusFix()` 范围即可。
   - 疯狂点击后 `document.activeElement` 是否指向 INPUT？→ 是 = 平台层问题；否 = DOM 层问题。

### 🐞 深度修复：Blockly 全局事件劫持与 Electron 幽灵焦点漏洞 (Event Hijacking & Ghost Focus)

#### 1. 现象描述 (Phenomenon)
- 用户点击右侧 AI 助手的文本输入框时，虽然 UI 画面的 CSS (Focus ring) 发生变化证明被点击，但输入框**没有任何闪烁的光标**，且敲击键盘毫无反应（无法输入任何文字）。

#### 2. 核心机理 (Root Causes)
- **Blockly 贪婪劫持**: Blockly 引擎为了接管全局的积木拖拽和点击空白处取消选中，它在 `document` (整个界面的最外层) 上绑定了贪婪的 `pointerdown` / `mousedown` 监听器。点击界面的任何非屏蔽区域，事件都会冒泡到顶层被 Blockly 捕获，并直接无情调用了 `e.preventDefault()`。这**一招毙命摧毁了浏览器原生的“激活输入框、显示闪烁光标”的行为**。
- **Electron 平台焦点割裂**: 此外，在 Windows 上的 Electron 存在一定概率的 DOM 层面上选中了元素，但是操作系统系统层面的焦点窗口未激活，导致键盘事件被吃掉。 

#### 3. 终极应对策略 (Ultimate Mitigation)
- **事件冒泡切断 (Event Propagation Cutoff)**:
    - 在 `AiAssistantPanel.tsx` 的 `<textarea>` / `<input>` 等输入控件上绑定 `onMouseDown={(e) => e.stopPropagation()}`。这是一个“终极防火墙”，使得点击事件在输入框层级被完美消化与终止，**彻底切断了向上传递给 Blockly 的冒泡路径**。
- **系统焦点强制夺回 (Forced System Focus)**: 
    - 辅以 `window.focus()` 的调用，加上 `setTimeout(() => input.blur(); input.focus(), 0)` 的强行刷新光标魔法，从底层 OS 到浏览器 DOM 彻底修复此交互断层。

### [16:50] 代码编辑区“滚动跳转”现象深度诊断 (Editor Scrolling & Jump Analysis)

#### 1. 现象描述 (Phenomenon)
用户在左侧积木区添加/修改积木时，右侧 Monaco Editor 的滚动条偶尔会发生意外跳转（如回弹到顶部或偏移几行），即便编辑器窗口尺寸并未发生改变。

#### 2. 核心机理 (Root Causes)
- **全量更新开销 (Virtual DOM Sync)**: 
    - 积木变更触发 `AppInner` 渲染，通过 `setValue` 全量重写编辑器内容。
    - Monaco 在接收到新值时，若行数发生变动，会重新计算虚拟滚动高度。在 React 的 Reconciliation（调和）期间，如果布局发生微秒级的失稳，编辑器会执行一次“纠偏”。
- **布局敏感度 (Automatic Layout)**:
    - 开启 `automaticLayout: true` 后，编辑器会高频检测父容器尺寸。
    - 在 React 组件重绘瞬间，父容器可能会产生极短暂的“状态计算期”，导致 Monaco 误判定为尺寸变化并强制触发 `layout()`。
- **边界锚点误差 (Scroll Beyond Last Line)**:
    - 若此项设为 `true`，编辑器底部会生成动态空白区。
    - 在内容大幅变化时，滚动条的比例算法会因为“溢出空白”的重新计算而产生坐标映射误差，导致跳转。

#### 3. 应对措施 (Mitigation)
- **已实施**: 
    - 稳定了 `onCodeChange` 的函数引用（`useRef`），减少不必要的重组件调和。
    - 设置 `scrollBeyondLastLine: false` 以确定物理边界。
- **未来路线**: 考虑从 `setValue` 迁移为基于 `pushEditOperations` 的 Diff 增量更新，从根本上保留编辑器视图状态。

### STM32 Local Patch 编译链修复 (Local Patch Build Chain Fix)

- **对策**:

    - **工作流集成**: 将清理逻辑挂载至 `predev` 生命周期，实现零干预下的“一键自愈”。
