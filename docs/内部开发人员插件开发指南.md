## 1. 扩展积木 (Adding Custom Blocks)



**推荐结构**:
```text
src/modules/extensions/ (推荐新建此目录用于存放扩展)
└── my_sensor/               <-- 模块专属文件夹
    ├── index.ts             <-- 入口文件 (只负责导出模块定义)
    ├── blocks.ts            <-- 积木 UI 定义
    ├── generator.ts         <-- Arduino 代码生成逻辑
    ├── assets/              <-- 资源文件
    └── README.md            <-- 说明文档
```

### 1.2 开发步骤

1.  **创建目录**: 在 `src/modules/extensions/` 下新建你的模块文件夹（如 `my_sensor/`）。
2.  **编写积木**: 建议包含 `index.ts` (导出模块定义), `blocks.ts`, `generator.ts`。
3.  **自动注册 (Auto-Discovery)**: 
    > [!IMPORTANT]
    > **无需再次手动修改 `src/modules/index.ts`**。
    > 系统会自动发现 `src/modules/extensions/*/index.ts` 及其导出的模块对象。您只需重启 Dev Server (或等待 HMR) 即可在界面看到结果。

### 1.3 积木定义参考

- **Arduino Generator Helpers**: 请务必使用 `arduinoGenerator.addInclude`, `addVariable`, `addSetup` 等辅助方法。
- **结构化生成**: 系统现在内置了 [CodeBuilder](file:///c:/Users/wofy/Desktop/embedblocks-studio/src/generators/utils/CodeBuilder.ts) 引擎。它可以自动处理代码段落（Include, Macro, Global, Setup, Loop）的冲突和排序。
- **国际化 (I18n)**: **严禁硬编码 UI 字符串**。
    - **逻辑**: 所有积木标签、提示、下拉选项必须通过 `Blockly.Msg` 引用。由于源码级扩展是编译进主包的，词条需同步添加到 `src/locales/` 下的 `zh.json` 和 `en.json`。
    - **命名规范**: 建议使用 `BLOCKS_MYMODULE_TITLE` 格式的键名，防止命名冲突。
- **示例代码**: 见 `src/modules/examples/`。
    - **启用示例**: 示例模块默认不加载。需在 `src/modules/index.ts` 注册，并在 `src/config/toolbox_categories.ts` 配置工具箱显示位置。详见 `src/modules/examples/README.md`。
    - **独立模块的国际化 Trick**: 如果希望编写不依赖全局 `src/locales` 的“便携式”核心模块（如 `examples` 或 `extensions` 目录下的源码），可以在模块 `init` 函数中使用 `Object.assign(Blockly.Msg, ...)` 动态注入翻译。但这不支持动态语言切换（需刷新页面生效）。具体写法参考 `src/modules/examples/basic_led/blocks.ts`。

## 2. 核心架构逻辑 (Core Architecture)

项目在 2026-01 进行了深度重构，采用 **Context + Hooks** 的解耦模式。

### 2.1 状态管理与业务逻辑分离
核心 Context 现在仅负责存储状态和分发 Provider，具体的业务逻辑（I/O、IPC 通讯、算法）被提取到了专门的 Hooks 中：

| 模块 | 核心 Hook | 职责 |
| :--- | :--- | :--- |
| **文件系统** | `useProjectOps` | 处理项目创建、打开、保存、元数据同步。接收分组后的 `ProjectState` 对象。 |
| | `useAutoBackup` | 处理静默自动保存逻辑（防抖时间 3000ms）。 |
| **串口通讯** | `useSerialMonitor` | 统一订阅和分发后端 IPC 串口事件。在 `SerialContext` 中结合 `useReducer` 管理复杂状态。 |
| | `useSerialActions` | 封装所有底层的串口控制指令。 |
| **UI/Toolbox** | `useToolbox` | 核心解耦 Hook。封装了基于当前型号和语言的工具箱分类加载、过滤与刷新逻辑。 |

### 2.2 状态管理进化：useReducer 模式
对于具有大量关联参数的模块（如串口设置），我们采用了 `useReducer` 替代分散的 `useState`。

**优势**:
- **原子性**: 确保波特率、校验位等多个关联参数在同一个 Action 中更新，防止由于渲染异步导致的中间状态不一致。
- **可维护性**: 所有的状态修改逻辑都集中在 Reducer 函数中，易于追踪和调试。

### 2.3 独家机制揭秘 (Hidden Gems)

#### 2.3.1 安全沙箱机制 (The Sandbox)
为了安全地执行第三方插件的 JS 代码（积木定义和生成器），我们构建了一个隐形的 `iframe` 沙箱。这是项目安全性的基石。
- **入口**: `src/sandbox/sandbox.html`
- **逻辑**: `src/sandbox/sandbox-entry.ts`
- **通信**: 主窗口通过 `postMessage` 将 XML/JSON 发送给沙箱 -> 沙箱内的 Headless Blockly 解析并生成代码 -> `postMessage` 返回 C++ 代码。

#### 2.3.2 智能变量扫描 (Scope-Aware Variable Scanner)
位于 `src/utils/variable_scanner.ts` 的扫描器是编辑器的“大脑”。
- **功能**: 它不仅扫描变量名，还扫描**作用域**。
- **getContextAwareVariables**: 
    - 如果积木在 `Function Definition`内，它会从 `functionScopes` Map 中查找该函数的参数和局部变量。
    - 下拉菜单 (`FieldDropdown`) 会动态调用此函数，从而实现“只显示当前可见变量”的高级特性。这是普通 Blockly 项目不具备的高级特性。

### 2.2 代码生成引擎 (CodeBuilder)
为了解决复杂的库依赖和 Setup/Loop 初始化冲突，项目引入了 `CodeBuilder` 机制。

**优势**:
1. **防重性**: 自动去重重复的 `#include` 语句。
2. **有序性**: 严格保证宏定义 -> 类型定义 -> 全局变量 -> 函数原型的生成逻辑顺序。
3. **注入性**: 支持在任意积木中通过 `addSetup` 或 `addLoop` 注入初始化代码，无需关心主函数的结构。

**使用示例**:
```typescript
// 在 generator.ts 中
arduinoGenerator.addInclude('sensor_lib', '#include <MySensor.h>');
arduinoGenerator.addSetup('sensor_init', 'mySensor.begin();');
arduinoGenerator.addVariable('sensor_obj', 'MySensor mySensor(1, 2);');
```

## 3. 扩展芯片支持 (Adding Custom Support for Boards)

芯片配置不再建议修改核心文件，而是采用**外挂配置**的方式。

> [!IMPORTANT]
> **开发板配置的三种路径** (v1.2.0 更新):
> 1. **JSON 快速添加** (推荐): 在 `src/data/boards/custom/` 目录下创建 `.json` 文件，重启即生效，无需编译。
> 2. **TypeScript 编译**: 在 `src/config/custom/` 下编写 TypeScript 对象，直接编译进软件。
> 3. **外部插件**: 在插件目录中使用 `board.json` 定义硬件（无需重新编译）。详见 [插件系统手册-硬件规范](./普通用户插件系统开发手册.md#81-标准硬件扩展目录结构)。

### 3.1 方案 A：单个自定义板子 (Single Custom Board)

适用于只需要添加一两个特定板子的情况。

1.  **定义**: 在 `src/config/custom/` 下创建 `.ts` 文件 (如 `example_board_c3.ts`)。
2.  **注册**: **系统会自动发现并将其归入“芯片实验室”家族**。如果您需要将其加入现有家族（如 Arduino），才需要手动修改对应的家族文件。

**开发板 I18n 规范**:
芯片名 (`name`) 和引脚标签 (`pinLabels`) 支持“双模”翻译：
- **模式 A (词条化)**: 在 `pinLabels` 中使用 `%{KEY}`，并在主语言包中定义。
- **模式 B (内置对象)**: 直接在对象中使用 `{zh: "...", en: "..."}`。系统会自动根据当前语言加载（推荐用于简单的芯片扩展，如示例中的 C3 SuperMini）。

```typescript
// src/config/custom/example_board_c3.ts
export const EXAMPLE_C3_SUPERMINI: BoardConfig = {
    id: "example_c3_supermini",
    name: { zh: "示例: C3 SuperMini", en: "Example: C3 SuperMini" }, // 推荐使用对象直写
    // ...
};
```

### 3.2 方案 B：大规模自定义系列 (Zero-Config Series) —— **推荐**

适用于需添加大量板子或希望实现“文件即注册”的情况（如教育套件系列）。我们利用 Vite 的 `glob` 功能实现了自动发现。

1.  **从哪里开始**: 查看 `src/config/custom/example_edu_series/` 目录。
2.  **如何添加**: 
    - 只需在该目录下创建 `board_xxx.ts`。
    - 或者创建 `board_yyy/index.ts` 文件夹。
    - **无需写任何注册代码**。系统在启动时会自动扫描所有此类文件，并将其展示在“芯片实验室 (Chip Lab)”分类下。

### 3.3 综合注册示例 (Integration)

可以将上述两种方法混合使用。修改家族文件 `src/config/esp32_boards.ts` 如下：

```typescript
// 1. 引入单个板子 (方案 A)
import { EXAMPLE_C3_SUPERMINI } from './custom/example_board_c3'; 
// 2. 引入整个系列 (方案 B)
import { EDU_KIT_SERIES } from './custom/example_edu_series';

export const ESP32_SERIES: BoardSeries[] = [
    // ... 原有的官方系列 ...
    {
        id: 'esp32_generic',
        name: 'Generic ESP32',
        boards: [
            ESP32_DEVKIT_V1,
            EXAMPLE_C3_SUPERMINI, // <--- 方案 A：加入现有列表
        ]
    },
    
    // ... 
    
    EDU_KIT_SERIES // <--- 方案 B：作为一个独立的分组加入
];
```

通过这种方式，您可以灵活地管理不同规模的自定义硬件。

### 3.4 深度技术解析：自动发现机制 (Deep Dive: Auto-Discovery)

`src/config/custom` 目录之所以能实现“文件即插件”的神奇效果，核心依赖于 Vite 构建工具的 `import.meta.glob` 特性与我们编写的 **鸭子类型 (Duck Typing)** 扫描逻辑。

#### 1. 核心实现原理 (The Mechanism)

代码位于 `src/config/all_boards.ts`，其工作流程如下：

1.  **文件扫描 (Glob Import)**:
    ```typescript
    const customModules = import.meta.glob(['./custom/*.ts', './custom/*/index.ts'], { eager: true });
    ```
    -   `eager: true`: 告诉打包器在构建时直接导入这些模块（同步加载），而打入最终的 JS 包中。
    -   **扫描范围**: 所有 `custom` 下的 `.ts` 单文件，以及子文件夹下的 `index.ts`。

2.  **智能识别 (Duck Typing)**:
    代码并不关心你导出的变量名是什么，而是动态检查导出对象的**形状**：
    -   如果对象包含 `boards` 数组 -> 判定为 **系列 (Series)**。
    -   如果对象包含 `mcu` 和 `pins` 属性 -> 判定为 **单板 (Board)**。

#### 2. 数据流转 (Data Flow)

1.  **收集**: 扫描到的单板会被自动归纳到 `dev_single` ("单板仓库") 系列中；扫描到的系列则保持独立。
2.  **组装**: 它们共同组成了 `CUSTOM_FAMILY` ("芯片实验室")。
3.  **注册**: `BoardRegistry` 在初始化时遍历所有家族，将这些实验室板子与官方 Arduino/ESP32 板子一视同仁地注册到系统中。

这种零侵入的设计，确保了每位开发者可以拥有自己的本地测试硬件而不干扰主线代码，同时也支持了 Git Submodule 的无缝集成。

## 4. 插件系统开发 (Plugin System)

如果您希望开发**不需要重新编译软件**即可分发的插件（适合最终用户），请参考独立的插件开发指南：

👉 **[插件系统深度手册 (Master Manual)](./普通用户插件系统开发手册.md)**

该手册已全面升级，核心亮点包括：
- **深度国际化支持**: 支持 `%{KEY}` 占位符、`%{BKY_...}` 全局词条引用。
- **智能回退机制 (Smart Fallback)**: 
    - 插件加载时，系统会按以下顺序查找翻译：
    1. `Blockly.Msg[KEY]` (Blockly 标准词条)
    2. `translations[KEY]` (插件自带的 locales/*.json)
    3. `Value` (原始字符串作为默认值)

- **双模翻译方案**: 支持“词条引用”与“对象直写”两种模式，大幅简化了轻量级硬件插件的开发流程。
- **技术底层解密**: 全面还原了 C++ 库集成环境变量、生成器 API 细节及硬件兼容性逻辑。
- **示范工程全量覆盖**: 所有官方示例均已根据最新标准完成 I18n 适配。

## 5. 工具箱定制化逻辑 (Toolbox Customization)

EmbedBlocks 支持用户在设置界面手动隐藏/显示特定工具箱分类。

### 5.1 过滤原理 (Filtering Mechanism)

工具箱的过滤发生在 `src/App.tsx` 的 `refreshToolbox` 函数中：
1.  **加载板级配置**: 调用 `BoardRegistry.getToolboxConfig(selectedBoard)` 获取初始分类列表。
2.  **获取用户偏好**: 从本地存储加载 `hiddenCategories`（分类 ID 字符串数组）。
3.  **执行过滤**: 遍历 `contents`，检查 `item.id || item.name` 是否在隐藏列表中。

### 5.2 动态同步 (Dynamic Sync)

设置中心 (`ToolboxSettings.tsx`) 的列表是**非硬编码**的：
- 它会实时调用 `BoardRegistry` 获取当前开发板的实际工具箱定义。
- 这保证了设置界面呈现的分类与左侧侧边栏看到的完全对应。

### 5.3 特殊分类：Dev Test (红色标记)

在开发板配置中，您可能会看到一个带有红色图标的 **Dev Test** 分类。
- **含义**: 这是系统预留的开发调试分类（定义在 `BoardRegistry.ts` L126）。
- **用途**: 专门用于开发者验证新开发的积木，不建议普通用户开启。
- **移除**: 若需正式发布，可在 `BoardRegistry.generateToolbox` 中将其代码行删除或注释。

- 搜索字典归一化：若新增积木涉及特定的中英文对应关系，请同步维护 `src/config/search_dictionary.ts`。

## 6. 开发者偏好与记忆 (Developer Preferences)

项目支持通过本地配置文件自定义开发体验，这些文件已被加入 `.gitignore`。

- **`DEV_PATHS.json`**: 存储本地工具的绝对路径（如 `git.exe`），解决跨环境路径识别问题。
- **`DEV_PREFERENCES.json`**: 存储个人习惯偏好（语言、注释风格、I18n 强制等级）。

### 2.2 协作契约 (Tacit Agreement)

所有开发者均需遵守 **[协作默契指南](./协作公约.md)**，核心准则包括：
-   **中文优先**: 对话、反馈及文档默认使用简体中文。
-   **代码自注释**: 修改逻辑时必须主动添加中文注释。
-   **I18n First**: **严禁硬编码任何 UI 字符串**。新增功能必须同步维护 `src/locales/` 下的中英文 JSON 词条库。所有开发板定义的 `name` 和 `pinLabels` 必须支持中英双语。

## 7. UI 设计范式与沉浸式交互 (UI Design Patterns)

EmbedBlocks 追求“工业级”的视觉美感。在开发新 UI 时应遵循以下规范：

### 7.1 沉浸式覆盖层 (Immersive Overlays)
对于大型信息展示（如“关于”页面），推荐使用 **AboutOverlay 模式** 替代传统弹窗：
- **视觉控制**: 使用 `backdrop-filter: blur(12px)` 实现毛玻璃背景。
- **动效**: 结合 Tailwind 的 `animate-in fade-in` 和 `zoom-in-95` 提升流畅度。
- **状态管理**: 统一挂载在 `App.tsx` 的 `GlobalListeners` 中，通过 `UIContext` 驱动状态。

### 7.2 Markdown 渲染规范
系统内置了 `MarkdownRenderer` 组件。在展示文档时：
- **层级管理**: 确保父容器具备 `prose prose-invert` 类名以降级渲染标准 Markdown 样式。
- **自定义滚动**: 使用 `custom-scrollbar` 保持与 IDE 整体风格一致。

## 8. 系统桥梁扩展 (IPC Bridge)

若需新增与操作系统交互的功能（如文件操作、外部程序调起）：

### 8.1 路径传递原则
在读取项目或帮助文档时，IPC 接口应采用 **(Content, Path)** 对象模式返回。这允许前端在需要时执行“在外部打开”或“跳转至目录”等二次操作。

### 8.2 安全调起 (Shell Bridge)
通过 `shell:open` 统一处理外部路径打开请求。前端应通过 `window.electronAPI.openExternal(path)` 安全调用，严禁在渲染进程直接操作 `shell` 模块。

## 9. FAQ

**Q: 为什么积木不能像芯片那样自动发现？**
A: **现在已经支持自动发现！** 放在 `src/modules/extensions/` 下的模块（需导出 `index.ts`）将被自动扫描并注册。

**Q: 我修改了 `platformio.ini` 相关的字段，需要重置项目吗？**
A: 是是的。构建配置缓存在项目中。修改板子定义后，建议并在 IDE 中执行 "Clean Build" 或重新生成配置文件。

**Q: 启动开发服务器时提示 `EACCES: permission denied` 怎么办？**
A: 这是因为 Windows 可能保留了该端口范围（由 Hyper-V 或内部服务使用）。项目已默认改用 `5100` 端口并开启了自动端口搜寻（`strictPort: false`）。如果依然报错，系统会自动尝试下一个可用端口。

**Q: 为什么新启动的应用无法打开或提示“单实例限制”？**
A: 如果之前的应用实例没有正常退出，可能会残留后台进程。我们提供了 `npm run dev` 自动清理机制。您也可以手动运行 `node scripts/cleanup.js` 来强制终止所有残留的 Electron 进程。

**Q: 为什么可以修改核心代码（如 `toolbox_categories.ts`）后，当前打开的项目被清空了？**
A: 这是 Vite 热更新 (HMR) 带来的副作用。当核心配置文件发生变更时，Blockly 可能会被强制卸载并重新初始化。如果此时状态同步出现时序问题（Race Condition），“空项目”状态可能会被自动备份机制捕获并覆盖临时文件。
> [!WARNING]
> **开发建议**: 在修改核心架构或配置文件时，请务必先**关闭当前正在编辑的重要项目**，或使用无关紧要的测试项目进行调试。

## 11. 底层服务细节 (Low-Level Services)

### 11.1 PlatformIO 编译服务 (PioService)
`electron/services/PioService.ts` 实现了从零配置到编译的全流程：
- **便携模式 (Zero-Config)**:

2. **translations[KEY]**: 其次查找插件自带的 `locales/*.json`。
3. **Value Literal**: 最后回退到原始字符串。

`ExtensionRegistry` 内置了状态保护机制。当 `i18n` 语言变更事件触发时：
1. 系统自动从 `originalManifest` 备份中恢复原始的 `%{KEY}` 占位符。
2. 使用新语言环境重新执行翻译逻辑。

## 10. 扩展市场系统 (Extension Marketplace)

EmbedBlocks 引入了自主管理的扩展市场系统，允许用户分发和安装插件。

👉 **[扩展市场发布指南 (Master Manual)](./插件市场发布指南.md)**

### 10.1 市场源配置
用户可以在 `ExtensionsModal` 的 Marketplace 选项卡中管理市场源。系统支持：
- **JSON 清单 URL**: 直接指向 `marketplace.json` 的链接。
- **GitHub 仓库链接**: 系统会自动将其归一化为 `raw.githubusercontent.com` 下的 `marketplace.json`。

### 10.2 开发者发布规范
若要使您的仓库支持市场发现，请在仓库根目录（或指定路径）下放置 `marketplace.json`：
```json
{
  "extensions": [
    {
      "id": "my-cool-plugin",
      "version": "1.0.0",
      "name": "Cool Plugin",
      "description": "This does amazing things.",
      "author": "YourName",
      "downloadUrl": "https://github.com/user/repo/archive/refs/tags/v1.0.0.zip",
      "icon": "https://raw.githubusercontent.com/user/repo/main/icon.png"
    }
  ]
}
```

### 10.3 安装生命周期
1. **发现**: `MarketplaceService` 异步获取所有已注册源的清单并合并显示。
2. **下载**: 用户点击安装后，系统下载指定 `downloadUrl` 的 ZIP 包到临时目录。
3. **分发**: 自动识别 ZIP 内部包含 `manifest.json` 的根目录，并调用 `ExtensionService.importExtension` 进行解压、移动和资源注入。
4. **实时生效**: 安装完成后，系统会自动刷新工具箱，无需重启即可开始使用。

### 10.4 智能版本控制 (Smart Versioning)

系统内置了基于语义化版本 (Semver) 的保护机制：

-   **Update (更新)**: 当市场版本 > 本地版本时，显示绿色“更新”按钮。
-   **Downgrade (降级)**: 当市场版本 < 本地版本时，显示红色“降级”按钮。
-   **Overlap (覆盖保护)**: 任何降级操作（包括本地导入旧版 ZIP）都会触发二次确认弹窗。仅在用户明确同意后，系统才会通过 `force: true` 参数执行强制覆盖。

## 11. 双仓库协作工作流 (Dual-Repo Workflow)

为了实现插件生态与主程序解耦，项目采用了 **Git Submodule** 架构。

-   **主仓库**: `embedblocks-studio` (不直接包含插件源码，仅包含指针)
-   **插件仓库**: `blocksLib` (独立维护，作为子模块挂载)

### 11.1 常用脚本 (Utility Scripts)

不管是更新主程序还是发布新插件，请统一使用我们提供的自动化脚本：

```powershell
.\scripts\sync_repos.ps1
```

该脚本提供交互式菜单：
1.  **仅更新 blocksLib**: 自动打包 ZIP -> 提交子仓库 -> 推送。
2.  **仅更新 主项目**: 提交主项目 -> 推送。
3.  **联合更新 (推荐)**: 
    - 先更新子仓库 (打包+推送)
    - 自动将主项目中的子模块指针 (`git add blocksLib`) 更新到最新 Commit。
    - 提交并推送主项目。
    - **这是防止“游离指针”导致版本不同步的最佳实践。**

### 11.2 自动化打包与发布 (Auto-Packing & Publishing)

现在，`blocksLib/pack_plugins.ps1` 不仅仅是压缩工具，它是**全自动发布工具**。

**功能**:
1.  **自动压缩**: 将每个插件文件夹打包为 `.zip`。
2.  **自动索引**: 读取每个插件的 `manifest.json`。
3.  **自动生成**: 覆盖 `marketplace.json`，自动填充 Version, Description, Icon, DownloadUrl (指向 GitHub Raw)。

**这意味着您无需手动编辑 `marketplace.json`。** 只要维护好每个插件自己的 `manifest.json`，发布工作就是一键完成。

## 12. 标准模块架构 (Standard Module Architecture - SMA)

为解决复杂积木（如 AI/IoT）单文件代码过长、UI与逻辑耦合过重的问题，自 2026-01 起，推荐使用 **SMA 目录结构** 开发核心模块。

### 12.1 目录规范
```text
src/modules/core/tinyml/    <-- 模块根目录
├── index.ts               <-- [Entry] 模块入口，负责导出 BlockModule 对象
├── blocks.ts              <-- [View] 纯粹的 Blockly 积木 UI 定义 (init, mutator)
└── generator.ts           <-- [Controller] 纯粹的 C++ 代码生成逻辑
```

### 12.2 优势
1.  **高内聚**: 将 "长什么样" (`blocks.ts`) 和 "怎么工作" (`generator.ts`) 物理分离，互不干扰。
2.  **低耦合**: Generator 不再依赖 Blockly UI 类，便于在 Headless 环境（如 CI/CD 或沙箱）中独立加载运行。
3.  **可维护**: 单文件行数限制在 300 行以内，提升代码可读性。

**建议**:
所有超过 5 个积木或包含复杂 Mutator 的新模块，**必须** 采用此架构。
